# Mermaid Visualizer to Diagram Forge - Migration Strategy and Build Execution Guide

## Document status

This is the **working strategy document** for building Mermaid Visualizer into a diagram system **on top of the existing repository** (the "Diagram Forge" project).

This document is written as an instruction guide, not a summary.

### How to read this document
- Follow phases in order.
- Do not skip acceptance criteria.
- When a real choice is required, it is marked **Decision Gate**.
- Everything else is an instruction, not a suggestion.

---

## 1. Project directive

Build Mermaid Visualizer into the "Diagram Forge" app as a system with these properties:

- semantic source of truth (Domain-Specific Language and Abstract Syntax Tree)
- deterministic compilation and rendering
- immutable semantic artifacts
- versioned presentation overlays
- Application Programming Interface first architecture
- web interface as a client of services
- WYSIWYG-style canvas editing only for presentation overlays

### Mandatory interpretation of the canvas editor
The canvas editor is part of the product, but it is **not** the semantic editor.

The canvas editor may:
- position elements visually
- annotate
- group
- highlight
- redact
- frame

The canvas editor may not:
- create or delete semantic nodes
- create or delete semantic edges
- change semantic meaning
- change canonical Mermaid output
- change semantic hashes

## 2. Baseline assumption and migration mode

This strategy assumes we are building on the existing repository.

### Why this is the default
The existing repository already provides useful assets (for example, user interface shell, authentication, rendering, deployment, and plain-English generation). Reusing those assets accelerates delivery and preserves working value.

### Decision Gate 1 - Continue migration or abort to greenfield
Complete Phase 0 first. Only switch to a new repository if Phase 0 reveals that boundary separation is practically unachievable.

Switch to a new repository only if **two or more** of the following are true after Phase 0 inspection:
- rendering is deeply embedded in user interface components and cannot be wrapped cleanly
- state, persistence, and rendering are tightly coupled across many files
- adding service boundaries requires touching most screens immediately
- test setup is so brittle that compiler and renderer cannot be tested independently within a reasonable effort
- current module structure prevents any clear separation of semantic, build, and presentation layers

If fewer than two are true, continue migration in the existing repository.

Do not make this decision before Phase 0.

## 3. Non-negotiable architecture rules

Treat these as project law.

### Rule 1 - Three layers must remain separate

#### Semantic layer
Contains:
- Domain-Specific Language definitions and schemas
- parser
- Abstract Syntax Tree types and constructors
- structural validation
- semantic validation
- canonicalization
- Mermaid generation
- policy evaluation

This layer answers: **What does the diagram mean?**

#### Build layer
Contains:
- deterministic rendering
- Scalable Vector Graphics and Portable Network Graphics generation
- output hashing
- render cache

This layer answers: **How do we render the same semantic meaning reproducibly?**

#### Presentation layer
Contains:
- canvas overlays
- layout overrides
- annotations
- group boundaries
- highlights
- viewport framing
- redaction elements

This layer answers: **How do humans present the built artifact for review and communication?**

### Rule 2 - Allowed direction of data flow
Allowed:
- semantic layer -> build layer -> presentation layer

Forbidden:
- presentation layer mutating semantic layer
- build layer mutating semantic layer
- user interface inventing semantic rules not enforced server-side

### Rule 3 - Mermaid is a target format
Mermaid is generated from canonical semantic representation.

Mermaid is never the primary source of truth for new diagrams.

### Rule 4 - Determinism and immutability are product features
Determinism and immutability are not implementation details. They are core product behavior and must be visible in:
- contracts
- tests
- logs
- user-facing history views

### Rule 5 - Service contracts own truth
The web client may provide user experience hints, but semantic truth, validation, canonicalization, hashing, and artifact persistence are defined by services.

## 4. Terminology used in implementation

Use these terms consistently in code, documentation, and tests.

- **DSL**: semantic input format authored by users or generated by other systems
- **AST**: internal semantic representation produced by the compiler
- **canonical DSL**: normalized and stable serialized DSL representation
- **canonical Mermaid**: Mermaid output generated deterministically from canonical AST
- **semantic artifact**: immutable record of a compiled and rendered diagram build
- **overlay**: immutable presentation-layer document linked to a semantic artifact
- **project**: logical grouping of related semantic artifacts
- **lineage**: parent-child history for artifacts or overlays
- **diagnostic**: structured message produced by validation or import steps
- **render config**: explicit rendering settings that affect output identity

## 5. Repository migration plan (existing repository only)

This section explains how to restructure the existing repository without breaking working functionality.

### 5.1 Migration pattern
Use a strangler migration pattern.

Instruction:
1. Keep the current application working.
2. Introduce new modules beside existing code.
3. Route one capability at a time through new service contracts.
4. Remove old direct paths only after the replacement path is stable and tested.

Do not rewrite the entire application at once.

### 5.2 Target repository shape
Create the following top-level modules or packages in the existing repository. Use names compatible with your current tooling conventions.

- `apps/web`
- `services/compiler`
- `services/renderer`
- `services/registry`
- `packages/shared-types`
- `packages/test-fixtures`
- `docs/architecture`

If the repository does not support a monorepo layout yet, create equivalent folders with the same responsibility boundaries.

### 5.3 Immediate refactoring directive
Before building new features, stop adding logic directly into user interface components for:
- Mermaid generation
- semantic validation
- rendering internals
- persistence decisions

Wrap existing behavior behind interfaces in `services/*` or adapters that can later be replaced.

### 5.4 Contract-first development directive
For each new endpoint or service method:
1. define request shape
2. define response shape
3. define error model
4. define versioning rule
5. implement
6. test

Do not start with user interface wiring.

## 6. Phase 0 - Current repository inspection and boundary setup

### Phase goal
Create an accurate map of the current system and install architectural boundaries so future phases can be implemented safely.

### Why this phase is mandatory
Without Phase 0, the team will keep placing new logic into old code paths. That will preserve coupling and block the architecture.

### Deliverables
- `docs/architecture/current-state.md`
- new module folders and stubs
- adapter interfaces around current rendering and generation paths
- contribution note enforcing boundary rules

### Step-by-step instructions

#### Step 0.1 - Map current execution paths
Create `docs/architecture/current-state.md` and document, with file paths:

- where plain-English generation is invoked
- where Mermaid text is created or modified
- where diagram preview rendering happens
- where export logic runs
- where authentication is enforced
- where persistence/logging happens
- where editor state is stored and updated
- which components call rendering or generation functions directly

This document must include a small flow diagram or bullet flow for the current path.

#### Step 0.2 - Identify coupling points and assign temporary adapters
For each direct user interface dependency on generation or rendering logic, create an adapter interface.

Examples of adapter categories:
- `GenerationAdapter`
- `RenderPreviewAdapter`
- `ArtifactBuildAdapter` (stub at this stage)

The adapter may initially call the old implementation internally.

The goal is to move user interface components onto interfaces before replacing internals.

#### Step 0.3 - Create new service module skeletons
Create module folders with README stubs describing responsibilities:
- compiler
- renderer
- registry
- shared contracts
- fixtures

Each README must contain:
- responsibilities
- non-responsibilities
- dependencies allowed
- dependencies forbidden

This prevents architectural drift later.

#### Step 0.4 - Add a contribution rule for new diagram logic
Add a short rule in your contribution documentation:

> New semantic, rendering, registry, and overlay logic must be implemented in service modules or adapters. User interface components may consume contracts and call adapters only.

#### Step 0.5 - Create a migration issue list
Create a tracked list of old code paths that must be replaced. Each item should include:
- current path
- replacement module
- status (`not started`, `adapter added`, `migrated`, `removed`)
- risk notes

### Acceptance criteria for Phase 0
- current-state architecture is documented with file references
- user interface rendering/generation paths are behind adapters in at least one place
- service module skeletons exist
- contribution rule is written
- migration issue list exists

Do not begin Phase 1 until all five criteria are complete.

## 7. Phase 1 - DSL v1 and compiler foundation

### Phase goal
Establish semantic authority and deterministic canonical Mermaid generation for one diagram family.

### Scope instruction
Implement one diagram family first. Use flowcharts as the first family unless there is a stronger product reason to choose another family.

Do not expand to multiple families until Phase 1 acceptance criteria are met.

### Deliverables
- DSL v1 specification document
- structural schema validation
- AST types and parser
- semantic validation
- canonicalization rules and implementation
- canonical Mermaid generation
- compiler contracts
- compiler tests
- compiler command line interface (recommended and treated as in-scope)

### Step-by-step instructions

#### Step 1.1 - Write the DSL v1 specification before coding
Create `docs/architecture/dsl-v1-flowchart.md`.

This document must define:
- DSL version field
- diagram family identifier
- node object schema (ids, labels, kinds, metadata)
- edge object schema (from, to, kind, label, metadata)
- optional style declarations (semantic only)
- optional constraints field
- identifier naming rules
- supported enumerations
- unsupported features list
- examples:
  - valid minimal document
  - valid complete document
  - invalid structural example
  - invalid semantic example

Instruction:
Keep the DSL semantic. Do not embed arbitrary Mermaid snippets except as a clearly marked temporary import escape hatch if absolutely required.

#### Step 1.2 - Implement structural schema validation
In `services/compiler`, implement schema validation for DSL v1.

Requirements:
- reject unknown fields unless intentionally allowed
- return structured errors
- include field path or pointer
- include error code and message
- separate schema errors from later semantic errors

Create fixture tests in `packages/test-fixtures` for:
- valid inputs
- invalid structure
- boundary cases (missing fields, wrong types, invalid enum values)

#### Step 1.3 - Define AST types
Define AST types as the internal semantic representation. These types must be explicit and stable.

Instructions:
- include normalized node and edge representations
- include explicit defaults after parsing
- avoid nullable ambiguity when a default can be made explicit
- include metadata slots if needed for future policy evaluation
- include source reference fields only if useful for diagnostics

The AST is not a mirror of DSL text. It is a compiler representation.

#### Step 1.4 - Implement parser from validated DSL to AST
Parser responsibilities:
- accept validated DSL v1
- construct AST
- fill explicit defaults in one place
- preserve source references for diagnostics if implemented
- never generate Mermaid directly

Do not mix semantic validation inside parser code. Keep parser and validator separate.

#### Step 1.5 - Implement semantic validation
Implement semantic validation as a separate pass over the AST.

Minimum rules for flowcharts:
- duplicate node ids (error)
- edge references unknown node ids (error)
- invalid edge kind for selected diagram family (error)
- duplicate edge identity if disallowed (warning or error, define rule)
- self-loop rules (define behavior and enforce)
- disconnected nodes or subgraphs (warning)
- cycle detection behavior (define behavior and enforce)
- reserved ids or naming convention violations (warning or error as defined)

Each diagnostic must include:
- code
- stage (`schema`, `parser`, `semantic`, `import`)
- severity
- message
- location (field path, node id, edge id, or source reference)
- machine-readable metadata when useful

#### Step 1.6 - Define canonicalization rules in writing
Before implementing canonicalization, create `docs/architecture/canonicalization-v1.md`.

This document must define:
- stable sort order for nodes
- stable sort order for edges
- label normalization rules
- metadata normalization rules
- default materialization rules
- serialization format rules for canonical DSL and AST
- hash input rules (exact byte representation to hash)

This step is mandatory. Do not code canonicalization without a written rule set.

#### Step 1.7 - Implement canonicalization and hashing
Implement canonicalization according to the written rules.

Compute hashes for:
- canonical DSL
- canonical AST serialization
- canonical Mermaid (after generation)
- compiler version identifier (metadata, not a content hash)

Use tests to prove:
- semantically equivalent DSL variants produce identical canonical DSL, AST, Mermaid, and hashes
- cosmetic input differences do not affect canonical outputs

#### Step 1.8 - Implement Mermaid generation from canonical AST
Generate Mermaid only from canonical AST.

Requirements:
- deterministic ordering
- deterministic formatting
- no random identifiers
- no presentation-layer layout data
- explicit and consistent escaping/quoting

Add fixture tests with expected canonical Mermaid strings.

#### Step 1.9 - Define and implement compiler endpoint contract
Create a compiler endpoint or service method such as `POST /compile`.

Response must include:
- compile status
- canonical DSL
- AST (full or summary plus optional full)
- canonical Mermaid
- diagnostics
- hashes
- compiler metadata (version, DSL version)

Keep the response explicit. Do not return loosely structured objects.

#### Step 1.10 - Implement compiler command line interface
Add a small command line interface that:
- reads a DSL file
- runs compile
- prints diagnostics
- optionally writes canonical Mermaid and JSON outputs

This tool is required for fast iteration, fixture generation, and debugging without the user interface.

### Acceptance criteria for Phase 1
- DSL v1 specification exists and is used by implementation
- compiler runs independently of user interface
- structured and semantic diagnostics are returned predictably
- canonical outputs are stable across repeated runs
- fixture tests prove canonicalization behavior
- one diagram family is fully supported through compile stage

Do not begin Phase 2 before all criteria are met.

## 8. Phase 2 - Deterministic renderer and render cache

### Phase goal
Render canonical Mermaid into deterministic SVG and PNG outputs with observable cache behavior.

### Deliverables
- renderer contract
- deterministic render implementation
- render config canonicalization
- output hashing
- render cache
- renderer tests

### Step-by-step instructions

#### Step 2.1 - Define renderer input and output contracts
Create a renderer contract in shared types.

Input must include:
- canonical Mermaid
- render config
- optional cache policy flags

Output must include:
- cache hit boolean
- SVG and PNG payloads or object keys (define mode)
- output hashes
- renderer metadata
- render config hash

Document which fields affect output identity.

#### Step 2.2 - Define render config v1
Create `docs/architecture/render-config-v1.md`.

Include:
- theme
- background
- width and height or viewport policy
- scale
- output format options
- defaults
- canonicalization rules

This prevents accidental output changes later.

#### Step 2.3 - Implement renderer as an isolated service module
The renderer module must:
- accept canonical Mermaid only
- apply canonicalized render config
- produce SVG and PNG
- compute hashes
- return metadata

The renderer must not:
- parse DSL
- generate Mermaid
- perform semantic validation

#### Step 2.4 - Control non-determinism explicitly
Pin and record:
- runtime version
- Mermaid version
- renderer implementation version
- render config version

If exact SVG bytes vary due to environmental differences, implement SVG normalization for hashing and document it.

#### Step 2.5 - Implement render cache
Cache key must include at least:
- Mermaid hash
- render config hash
- renderer version identifier if renderer behavior may change over time

Cache behavior:
- on hit: return stored output and mark `cache_hit = true`
- on miss: render, store, return, mark `cache_hit = false`

#### Step 2.6 - Add renderer tests and fixtures
Tests must prove:
- same Mermaid and config -> same hashes
- changed config -> changed hash when expected
- cache hit path returns correct outputs
- cache miss path stores outputs correctly

### Acceptance criteria for Phase 2
- renderer works independently of user interface
- deterministic behavior is documented and tested
- cache behavior is visible in responses and logs
- output hashes are stable for repeated runs

Do not begin Phase 3 before all criteria are met.

## 9. Phase 3 - Registry for immutable semantic artifacts and versioned overlays

### Phase goal
Persist semantic artifacts and overlays with separate histories and reproducible retrieval.

### Deliverables
- semantic artifact storage schema and service
- overlay storage schema and service
- object storage layout for rendered outputs
- artifact and overlay lineage queries
- registry endpoints

### Step-by-step instructions

#### Step 3.1 - Define semantic artifact model in writing
Create `docs/architecture/artifact-model-v1.md` describing all semantic artifact fields and immutability rules.

Minimum fields to define:
- artifact id
- project id
- created timestamp
- created by
- parent artifact id
- source kind (`dsl`, `mermaid_import`, `generated_draft`, etc.)
- canonical DSL
- canonical AST
- canonical Mermaid
- diagnostics
- hashes
- render config
- renderer metadata
- object keys for SVG and PNG
- status
- labels or tags (if supported)

Define which fields are immutable and which, if any, may be updated.

#### Step 3.2 - Define overlay model in writing
Create `docs/architecture/overlay-model-v1.md`.

Define:
- overlay id
- artifact id
- parent overlay id
- overlay payload sections
- overlay canonicalization and hash
- immutability rules
- optional labels or names for overlays

The overlay model must be separate from semantic artifact model.

#### Step 3.3 - Implement storage schema
Implement storage for:
- semantic artifacts
- artifact lineage or relationships
- overlays
- overlay lineage
- optional tags/labels
- audit metadata (if in scope)

Instruction:
Store rendered SVG and PNG in object storage using content-addressed keys, not mutable filenames.

#### Step 3.4 - Implement registry service methods
Implement methods and endpoints for:
- create semantic artifact
- get semantic artifact by id
- list artifacts by project
- get artifact lineage
- create overlay
- get overlay
- list overlays for artifact
- get overlay lineage

#### Step 3.5 - Implement retrieval composition methods
Add service methods that can fetch:
- semantic artifact alone
- semantic artifact plus selected overlay
- semantic artifact plus latest overlay (optional convenience behavior)

Keep this explicit so clients can choose the right representation.

### Decision Gate 2 - Store failed compile attempts in registry or outside registry
Make this decision before wiring user interface save behavior.

Recommended default:
- store successful compile-and-render outputs as semantic artifacts
- store failed drafts outside semantic artifact history (for example, draft sessions)

If you choose to store failed compile attempts in registry, create a separate model or status class so immutable successful history is not cluttered with transient drafts.

### Acceptance criteria for Phase 3
- semantic artifacts can be created and retrieved reproducibly
- overlay documents can be created and retrieved independently
- artifact and overlay histories are separate and navigable
- object storage uses content-addressed keys
- registry contracts are documented

Do not begin Phase 4 before all criteria are met.

## 10. Phase 4 - Web client integration for compile, render, and artifact workflows

### Phase goal
Move user workflows onto the compiler, renderer, and registry while keeping the existing application usable.

### Deliverables
- compiler-driven authoring screen
- render/build actions wired to services
- artifact history views
- artifact detail view
- Mermaid import path in user interface
- plain-English generation routed through compiler/import path

### Step-by-step instructions

#### Step 4.1 - Implement compiler-driven authoring mode
Create or update the main authoring screen so that:
- the primary editor is DSL-oriented
- compile is explicit or debounced
- diagnostics panel shows schema and semantic diagnostics
- generated Mermaid is shown as derived output

Instruction:
The user interface must make it visually clear that Mermaid is generated output, not the source of truth for new diagrams.

#### Step 4.2 - Add explicit stage actions
Provide separate actions in the interface during migration:
- Compile
- Render Preview
- Build Artifact

This separation helps debugging and user trust during migration. You can consolidate later once stable.

#### Step 4.3 - Add artifact history and detail screens
Implement:
- project artifact list
- artifact detail page
- lineage navigation

Artifact detail page must show:
- canonical Mermaid
- diagnostics summary
- hashes
- render metadata
- rendered outputs
- source kind
- parent artifact reference

#### Step 4.4 - Add Mermaid import path in user interface
Allow users to paste Mermaid and submit it to import/compile path.

Requirements:
- show diagnostics for unsupported or lossy imports
- preserve user visibility into what was accepted and normalized
- do not silently rewrite complex Mermaid without diagnostic messaging

#### Step 4.5 - Route plain-English generation through the new pipeline
Keep the existing plain-English feature, but rewire it:

Required flow:
1. user enters plain English
2. generation produces draft Mermaid or DSL
3. draft is sent to compile/import endpoint
4. diagnostics are shown
5. user refines
6. build artifact is created through registry path

Do not allow plain-English generation to bypass validation and artifact logic.

#### Step 4.6 - Remove direct rendering calls from user interface components
After the new render/build path is stable, replace old direct preview calls with renderer service calls through adapters or API client.

### Acceptance criteria for Phase 4
- users can build a semantic artifact end-to-end from the web client
- diagnostics are visible and actionable
- artifact history is browsable
- Mermaid import is available with diagnostics
- plain-English generation uses the same compile/build pipeline

Do not begin Phase 5 before all criteria are met.

## 11. Phase 5 - WYSIWYG-style canvas editor for presentation overlays

### Phase goal
Implement a visual editor for overlays and layout hints without changing semantic artifacts.

### Deliverables
- overlay schema implementation and canonicalization
- canvas mode on top of rendered SVG
- overlay save/load/history
- overlay selection in artifact views

### Step-by-step instructions

#### Step 5.1 - Finalize overlay schema and canonicalization before building canvas
Implement overlay canonicalization and hashing first.

Overlay sections should include:
- layout overrides (positions and layout hints only)
- annotations (text callouts, arrows, highlights)
- groups (visual boundaries)
- viewport framing
- optional style properties for overlays

The overlay payload must not contain semantic node creation/deletion operations.

#### Step 5.2 - Build canvas mode as a layered editor
Canvas mode must use:
- rendered SVG as locked base layer
- overlay objects as editable layer

Instruction:
Do not expose semantic editing tools in canvas mode. No add/remove semantic nodes. No edge creation. No Mermaid text editing in canvas mode.

#### Step 5.3 - Implement overlay persistence
Saving overlay changes must:
- create a new overlay version
- preserve prior overlay versions
- not trigger compiler
- not trigger renderer unless an explicit "export composited image" action exists
- not change semantic artifact hashes

#### Step 5.4 - Add overlay history and selection
Users must be able to:
- create multiple overlays for one artifact
- name overlays (optional but recommended)
- select overlay version
- compare overlays later

#### Step 5.5 - Add export behavior for composited view (optional in this phase)
If exporting composited images is implemented:
- treat it as presentation export
- do not store it as semantic artifact output
- attach export metadata to overlay or export history if needed

### Acceptance criteria for Phase 5
- canvas mode edits overlays only
- semantic hashes remain unchanged after canvas edits
- overlays are versioned and retrievable
- users can switch between overlay versions on the same artifact

Do not begin Phase 6 before all criteria are met.

## 12. Phase 6 - Diff engine for semantic artifacts and overlays

### Phase goal
Provide meaningful change explanations for both semantic and presentation histories.

### Deliverables
- semantic diff contract and implementation
- overlay diff contract and implementation
- user interface diff views
- summary and detail presentation

### Step-by-step instructions

#### Step 6.1 - Define diff result contracts
Create shared diff contracts for:
- semantic diff results
- overlay diff results

Include:
- summary counts
- categorized changes
- affected ids
- severity or impact classification if useful

#### Step 6.2 - Implement semantic AST diff first
Compare canonical ASTs and report:
- nodes added
- nodes removed
- nodes changed
- edges added
- edges removed
- edges changed
- metadata changes (if semantic)
- policy result changes (after Phase 7)

Treat semantic diff as the primary diff.

#### Step 6.3 - Implement overlay diff separately
Compare overlays and report:
- moved elements
- added/removed annotations
- group changes
- viewport changes
- style changes in presentation layer

Do not merge semantic and overlay diffs into one undifferentiated result.

#### Step 6.4 - Integrate diff views into history screens
Artifact history should offer semantic diffs.
Overlay history should offer overlay diffs.

Provide:
- readable summary first
- expandable raw detail second

### Acceptance criteria for Phase 6
- users can compare two artifacts and understand semantic changes
- users can compare two overlays and understand presentation changes
- diff results map to stable ids and structured contracts

Do not begin Phase 7 before all criteria are met.

## 13. Phase 7 - Policy bundles and governance checks

### Phase goal
Evaluate structured rules against the semantic layer and persist results with artifacts.

### Deliverables
- policy contract and result model
- policy evaluation engine against AST
- built-in policy bundles
- policy result persistence and display
- policy-aware semantic diffs

### Step-by-step instructions

#### Step 7.1 - Define policy rule and result contracts
A policy result must include:
- policy id
- policy bundle id (if applicable)
- pass/fail
- severity
- message
- affected nodes or edges
- machine-readable code
- evaluation metadata (version, timestamp if stored separately)

#### Step 7.2 - Implement policy evaluator against canonical AST
Run policies on AST only.

Do not evaluate policies on Mermaid text.

#### Step 7.3 - Implement policy bundle support
Bundle policies for different contexts (generic architecture, domain-specific, etc.) and allow bundle selection at build time or artifact evaluation time.

#### Step 7.4 - Persist and surface policy results
Attach policy results to semantic artifacts and surface them in:
- artifact detail
- history summaries
- semantic diff view (policy changes between artifacts)

### Acceptance criteria for Phase 7
- policy checks execute on semantic artifacts
- results are persisted and retrievable
- policy violations are visible and explainable in user interface
- policy changes appear in semantic diff when comparing artifacts

## 14. Mermaid ingestion strategy (explicit compatibility path)

This section governs Mermaid ingestion so it helps adoption without weakening semantics.

### 14.1 Purpose of Mermaid ingestion
Support existing Mermaid content from:
- repositories
- documentation systems
- user paste/import workflows
- generated outputs from plain-English tools or agents

### 14.2 Mermaid ingestion rules
Mermaid ingestion is a compatibility path.

Instruction:
- accept Mermaid input
- parse/normalize supported constructs
- produce diagnostics for unsupported constructs
- mark import as complete or partial
- preserve original Mermaid text when useful for traceability

Do not silently invent semantics for unsupported Mermaid constructs.

### 14.3 Decision Gate 3 - Import fidelity policy for unsupported Mermaid features
Choose one policy and record it in `docs/architecture/import-policy-v1.md`:

Option A (recommended default):
- partial import with diagnostics
- preserve original Mermaid
- allow build only for supported subset

Option B:
- reject unsupported constructs and require user edits before import

Make the policy explicit and consistent. Do not vary behavior by screen.

### 14.4 Migration instruction for existing plain-English generation
If the current repository already supports plain-English generation:
- keep it as an entry path
- route all outputs into import/compile flow
- show diagnostics
- require build artifact creation through registry path

Plain-English generation is a convenience input, not a privileged bypass.

## 15. Testing strategy (required from early phases)

### 15.1 Compiler fixture tests (start in Phase 1)
Maintain fixture sets for:
- valid DSL
- invalid structure
- invalid semantics
- canonical equivalence cases
- expected canonical Mermaid
- expected diagnostics
- expected hashes (when deterministic hash fixtures are stable)

### 15.2 Renderer determinism and cache tests (start in Phase 2)
Test:
- identical input and config produce stable outputs
- cache key behavior is correct
- config changes affect output identity as expected
- renderer metadata is recorded

### 15.3 Registry tests (start in Phase 3)
Test:
- create/get semantic artifact
- lineage traversal
- create/get overlay
- overlay lineage traversal
- immutability behavior
- object key content-addressing

### 15.4 End-to-end pipeline tests (start in Phase 4)
Test:
- DSL -> compile -> render -> artifact create -> retrieve
- Mermaid import -> compile/import -> render -> artifact create
- artifact + overlay retrieval composition

### 15.5 Canvas restriction tests (start in Phase 5)
Test that canvas workflows cannot:
- change semantic artifact payloads
- change canonical Mermaid
- change semantic hashes

This is a critical integrity test.

### 15.6 Diff and policy tests (Phases 6 and 7)
Test:
- semantic diff categories
- overlay diff categories
- policy results and policy changes in diffs

## 17. Instructions for coding assistants and future contributors

Add this section (or a shorter version) to contributor docs and optionally as a prompt for coding assistants.

### 17.1 Layer classification required before coding
Before implementing a change, classify it as one of:
- semantic
- build
- presentation
- registry
- orchestration
- user interface integration
- tests
- docs/contracts

If classification is unclear, stop and resolve it first.

### 17.2 Do not introduce cross-layer shortcuts
Reject implementations that:
- let canvas create semantic nodes or edges
- let renderer canonicalize raw user input
- let user interface define rules not enforced in compiler or policy engine
- let overlays modify semantic hashes

### 17.3 Preserve determinism
When changing canonicalization, rendering, or hash behavior:
- update tests intentionally
- update canonicalization docs
- record the change in architecture notes
- explain why hash changes are acceptable

### 17.4 Prefer explicit contracts over implicit objects
All endpoint and service payloads should use shared typed contracts, not ad hoc objects.

### 17.5 Record decision gates
When a Decision Gate is resolved:
- write the decision
- write the selected option
- write the impact on contracts and behavior
- commit it in `docs/architecture/decision-log.md`

## 18. Common implementation traps and prevention rules

### Trap 1 - Treating Mermaid as source of truth again
Prevention:
- keep DSL editor as primary path
- show Mermaid as derived
- enforce build flow through compiler

### Trap 2 - Expanding scope too early across diagram families
Prevention:
- finish one family fully through artifact creation before adding a second

### Trap 3 - Canvas becomes the primary editor
Prevention:
- enforce overlay-only schema and API restrictions
- test that semantic hashes remain unchanged after overlay saves

### Trap 4 - Mutable artifact records
Prevention:
- use append-only semantic artifacts and overlays
- separate mutable metadata if needed and define it explicitly

### Trap 5 - Hidden non-determinism from environment changes
Prevention:
- pin versions
- record renderer metadata
- run determinism tests after environment updates

### Trap 6 - Migration stalls because old paths remain easier
Prevention:
- keep migration issue list current
- route all new features into new modules only
- schedule explicit decommission tasks for old paths

## 19. Definition of done for the complete system

Mermaid Visualizer is complete when all statements below are true.

### Semantic and compile behavior
- diagrams are defined by DSL and AST for supported families
- canonical Mermaid is generated deterministically
- Mermaid import exists as a compatibility path with diagnostics

### Build behavior
- canonical Mermaid renders to deterministic SVG and PNG
- output hashes are stable and recorded
- render cache is active and observable

### Registry and history behavior
- semantic artifacts are immutable
- overlays are separate, immutable, and versioned
- semantic and overlay histories are browsable and reproducible

### User interface behavior
- users can author, compile, render, and build artifacts through service APIs
- users can import Mermaid with explicit diagnostics
- users can use a WYSIWYG-style canvas editor for overlays and layout hints
- canvas edits do not alter semantic artifacts or semantic hashes

### Review and governance behavior
- semantic and overlay diffs are available
- policy bundles can evaluate semantic artifacts
- policy results are visible and persisted

### Architecture integrity
- layer boundaries remain intact
- service contracts define truth
- tests protect canonicalization and determinism

At that point, the product is a diagram system with a web client, not just a diagram editor.
